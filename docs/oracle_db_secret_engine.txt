VAULT ORACLE DATABASE SECRET ENGINE - COMPLETE GUIDE

PURPOSE
The Oracle Database Secret Engine dynamically generates Oracle database credentials.
Instead of using static passwords, Vault creates a unique username and password for each
application request. These credentials automatically expire after a configured TTL.

WHEN TO USE ORACLE DATABASE SECRET ENGINE
- Application needs Oracle database credentials
- You want dynamic (rotating) database credentials instead of static passwords
- Enterprise Java application connects to Oracle DB
- Microservice needs Oracle DB access
- Any workload that queries an Oracle database

WHEN NOT TO USE ORACLE DATABASE SECRET ENGINE
- For non-Oracle databases (use appropriate database secret engine)
- For static API keys or passwords (use KV Secret Engine instead)
- For LDAP credentials (use LDAP Secret Engine instead)

HOW ORACLE DATABASE SECRET ENGINE WORKS
1. Vault connects to Oracle DB using a privileged service account
2. Application authenticates with Vault (using AppRole, Kubernetes Auth, etc.)
3. Application requests Oracle credentials from Vault
4. Vault dynamically creates a new Oracle user with a unique username and password
5. Vault returns the temporary credentials to the application
6. Application connects to Oracle DB using these credentials
7. After TTL expires, Vault automatically revokes the credentials from Oracle DB

BENEFITS OVER STATIC CREDENTIALS
- No shared passwords between applications
- Credentials automatically expire (no stale access)
- Full audit trail of who got which credentials and when
- Breach of one credential does not expose all applications
- No manual credential rotation needed

ORACLE DATABASE SECRET ENGINE ONBOARDING WORKFLOW

Step 1: Enable Database Secret Engine
Command: vault secrets enable database
Note: Only needed once per Vault cluster.

Step 2: Configure Oracle DB Connection
Vault needs a privileged Oracle account to create and revoke users.
Command:
vault write database/config/oracle-prod \
  plugin_name=vault-plugin-database-oracle \
  connection_url="{{username}}/{{password}}@oracle-host:1521/ORCL" \
  allowed_roles="*" \
  username="vault_admin" \
  password="vault_admin_password"

Parameters:
- plugin_name: Oracle plugin (must be installed on Vault server)
- connection_url: Oracle JDBC-style connection string
- username: Privileged Oracle account Vault uses to manage users
- password: Password for the privileged Oracle account

Step 3: Create Oracle DB Role
This defines what SQL Vault will run when creating temporary credentials.
Command:
vault write database/roles/payment-db-role \
  db_name=oracle-prod \
  creation_statements="CREATE USER {{name}} IDENTIFIED BY {{password}}; GRANT CONNECT TO {{name}}; GRANT SELECT ON payment_schema.orders TO {{name}};" \
  default_ttl=1h \
  max_ttl=24h

Parameters:
- db_name: Name of the database config created in Step 2
- creation_statements: SQL executed to create the temporary user
- default_ttl: How long credentials are valid
- max_ttl: Maximum credential lifetime

Step 4: Create Vault Policy
path "database/creds/payment-db-role" {
  capabilities = ["read"]
}
Command: vault policy write payment-db-policy payment-db-policy.hcl

Step 5: Attach Policy to Auth Method
Attach this policy to the AppRole or Kubernetes role of the application.

Step 6: Application Requests Credentials
Command: vault read database/creds/payment-db-role
Output:
  username = v-myapp-xkT9a2
  password = A1b2C3d4E5f6G7h8
  lease_duration = 1h

Step 7: Application Connects to Oracle
Application uses returned username and password to connect to Oracle DB.
Credentials automatically expire after TTL.

REQUIRED INPUTS FOR ORACLE SECRET ENGINE ONBOARDING
- Application Name
- Oracle DB Host and Port
- Oracle Service Name or SID
- Vault privileged Oracle account username and password
- Required Oracle grants (SELECT, INSERT, UPDATE etc.)
- Default TTL for credentials
- Environment (dev / qa / prod)
- Vault Namespace

ORACLE DB POLICY EXAMPLES

Read-only credential access:
path "database/creds/payment-db-role" {
  capabilities = ["read"]
}

Multiple DB roles:
path "database/creds/payment-db-role" {
  capabilities = ["read"]
}
path "database/creds/reporting-db-role" {
  capabilities = ["read"]
}

ORACLE DATABASE SECRET ENGINE BEST PRACTICES
- Use a dedicated privileged Oracle service account for Vault (not DBA or SYS)
- Grant only minimum required Oracle privileges in creation_statements
- Set short default_ttl (1h recommended for most applications)
- Set max_ttl to prevent indefinite access (24h maximum recommended)
- Rotate Vault's Oracle admin password regularly
- Monitor Oracle audit logs for Vault-created users
- Test credential revocation in lower environments before production
- Separate DB roles per application (do not share DB roles)
- Separate DB roles per environment (dev/qa/prod should have different Oracle connections)
- Never allow applications to renew credentials beyond max_ttl

LEASE RENEWAL AND REVOCATION
Renew credentials before expiry: vault lease renew database/creds/payment-db-role/<lease_id>
Revoke credentials immediately: vault lease revoke database/creds/payment-db-role/<lease_id>
Revoke all credentials for a role: vault lease revoke -prefix database/creds/payment-db-role/

ORACLE PLUGIN INSTALLATION NOTE
The Oracle database plugin requires the Oracle Instant Client libraries to be installed
on the Vault server. This is separate from the standard Vault installation.
Consult the Vault Ops team to confirm Oracle plugin is installed before onboarding.

TROUBLESHOOTING ORACLE DATABASE SECRET ENGINE

Error: Oracle connection failed
Cause: Wrong connection URL or Oracle DB unreachable from Vault server
Fix: Verify connection_url and network connectivity from Vault to Oracle host

Error: Insufficient privileges to create user
Cause: Vault's Oracle service account lacks CREATE USER privilege
Fix: Grant CREATE USER and required privileges to Vault's Oracle account

Error: Credential lease expired
Cause: Application held credentials beyond TTL
Fix: Application should request fresh credentials from Vault before expiry

Error: Plugin not found
Cause: Oracle Vault plugin not installed on Vault server
Fix: Contact Vault Ops team to install vault-plugin-database-oracle

Error: Permission denied reading database/creds
Cause: Policy path is wrong or not attached
Fix: Verify policy path is "database/creds/<role-name>"
