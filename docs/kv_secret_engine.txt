VAULT KV SECRET ENGINE - COMPLETE GUIDE

PURPOSE
The KV (Key-Value) Secret Engine is the most common Vault secret engine.
It stores static secrets such as API keys, passwords, tokens, and application configuration.

WHEN TO USE KV SECRET ENGINE
- Storing static API keys
- Storing application passwords
- Storing database passwords (static, not dynamic)
- Storing third-party service credentials
- Storing TLS certificates
- Storing application configuration values
- Any static secret that needs secure storage

WHEN NOT TO USE KV SECRET ENGINE
- For Oracle DB credentials that need to be dynamic (use Oracle Database Secret Engine)
- For LDAP credentials that need to be dynamic (use LDAP Secret Engine)

KV SECRET ENGINE VERSIONS
Version 1 (KV v1):
- Simple key-value storage
- No versioning of secrets
- Path: kv/myapp/config

Version 2 (KV v2) - RECOMMENDED:
- Supports secret versioning (keeps previous versions)
- Supports soft delete
- Supports metadata
- Path format: kv/data/myapp/config (note: "data" is added in path)

ALWAYS USE KV VERSION 2 IN ENTERPRISE ENVIRONMENTS.

KV SECRET ENGINE ONBOARDING WORKFLOW

Step 1: Enable KV v2 Secret Engine
Command: vault secrets enable -path=kv kv-v2
Note: Only needed once. Skip if already enabled.

Step 2: Create the Secret
Command:
vault kv put kv/payment-service/config \
  api_key="myapikey123" \
  db_password="securepassword"

Step 3: Verify Secret Created
Command: vault kv get kv/payment-service/config

Step 4: Create Policy for Application
path "kv/data/payment-service/*" {
  capabilities = ["read", "list"]
}

Step 5: Attach Policy to Auth Method (AppRole or Kubernetes Role)
This is done during auth method setup.

Step 6: Application Reads Secret
After authenticating, application reads the secret:
Command: vault kv get kv/payment-service/config

KV SECRET PATH NAMING CONVENTION
Format: kv/<application-name>/<environment>/<secret-name>

Examples:
- kv/payment-service/prod/config
- kv/user-service/dev/database
- kv/notification-service/qa/api-keys

REQUIRED INPUTS FOR KV ENGINE ONBOARDING
- Application Name
- Environment (dev / qa / prod)
- Secret Path
- Secret Keys and Values
- Namespace

KV SECRET ENGINE BEST PRACTICES
- Always use KV v2 (not v1) for versioning support
- Follow consistent path naming conventions
- Store one application's secrets under one path prefix
- Never store secrets in plaintext in code - always use Vault
- Enable secret versioning to support rollback
- Use destroy instead of delete for permanent removal
- Grant only read capability to application (not write/delete unless needed)
- Separate secrets per environment (dev/qa/prod)
- Regularly audit who has access to which KV paths
- Use Vault namespaces to separate team secrets

KV POLICY EXAMPLES

Read-only policy:
path "kv/data/payment-service/*" {
  capabilities = ["read", "list"]
}

Admin policy (for Vault ops team):
path "kv/data/payment-service/*" {
  capabilities = ["read", "list", "create", "update", "delete"]
}

Multi-application policy:
path "kv/data/payment-service/*" {
  capabilities = ["read"]
}
path "kv/data/shared/common-config" {
  capabilities = ["read"]
}

READING SPECIFIC VERSIONS (KV v2)
Read latest version: vault kv get kv/payment-service/config
Read version 2: vault kv get -version=2 kv/payment-service/config
List versions: vault kv metadata get kv/payment-service/config

KV SECRET ENGINE TROUBLESHOOTING

Error: Secret not found
Cause: Wrong path or secret not created yet
Fix: vault kv list kv/ to browse paths

Error: Permission denied on kv path
Cause: Policy uses wrong path format
Fix: KV v2 requires "kv/data/" prefix in policy, not just "kv/"
Correct: path "kv/data/myapp/*"
Wrong: path "kv/myapp/*"

Error: No value found at path
Cause: Secret was deleted or wrong key name
Fix: vault kv get to verify key names
