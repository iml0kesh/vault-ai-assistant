Vault Application Onboarding Guide

Purpose: This document explains the onboarding process for applications integrating with HashiCorp Vault.

When To Use:

New application requires secrets from Vault
Existing application migrating to Vault
Application requires secure credential management
Team wants centralized secret storage

Required Inputs:

Application Name
Environment (dev, qa, prod)
Namespace
Team Name
Secret Type
Authentication Method
Vault Path

Vault Onboarding Workflow:

Step 1: Consumer Raises Request The application team requests Vault onboarding.

Required information:

Application name
Environment
Secret requirement
Namespace

Step 2: Validate Request Vault team validates:

Namespace exists
Naming convention follows standards
Correct auth method selected
Required approvals exist

Step 3: Select Authentication Method Recommended authentication methods:

AppRole for VM-based applications
Kubernetes Auth for Kubernetes workloads
AWS IAM Auth for AWS-hosted workloads
JWT/OIDC for CI/CD pipelines

Step 4: Create Vault Policy Vault policy is created to control access.

Example policy:

path "kv/data/payment-service/*" { capabilities = ["read"] }

Step 5: Create Authentication Method Depending on use case:

AppRole created for VM applications
Kubernetes role configured for pods
IAM role mapped for AWS

Step 6: Share Integration Details Vault team shares:

Vault URL
Namespace
Role details
Secret path

Common Onboarding Questions:

Question: What is required before onboarding? Answer: Application details, namespace, auth method, and secret path.

Question: Which auth method should I use? Answer: Depends on workload type.

Question: Can onboarding happen without policy? Answer: No. Policy defines access control.
